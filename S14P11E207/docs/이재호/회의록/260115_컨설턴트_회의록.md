## 안건별 논의 내용

### 안건 1: 기획 방향성 확정

**논의 배경**

설문조사 168개 응답 기반 수정된 기획안에 대한 최종 승인 필요

**논의 내용**

- 조사 과정에서 기술 과제가 명확히 도출됨
- 먼저 프로토타입 제작 후 개선해 나가는 방식 권장
- 시연 발표 방식 고민 필요
- 빨리 제작 시 1반 vs 2반 대결로 화제성 및 피드백 확보 가능

**결정 사항 (Decision)**

✅ 기획 방향성 승인
✅ 프로토타입 우선 제작 후 이터레이션

**결정 근거**

설문 데이터 기반 사용자 니즈 검증 완료

### 안건 2: 기술 과제 정리

**논의 배경**

개발 단계 진입 전 기술적 리스크 사전 파악

**논의 내용**

- **보안**: 외부 노출, 캡처 방지 (가장 높은 리스크)
- **네트워킹**: 화상 연결 안정성
- **웃음 감지**: 실시간 감지 시 대역폭 이슈
- 캡처방지 기능: WebDRM(유료), 오픈소스 DRM, 알고리즘 보호
- 녹화 저장 장소 및 휘발 영상 선정 필요
- 로그인은 소셜 연동으로 간소화

**결정 사항 (Decision)**

✅ 개발 우선순위 확정
1. 화상 연결
2. 인증/로그인
3. 웃음 모델
4. 캡처/녹화 방지

✅ 설계 시 고려사항
- DRM 적용 방식
- 녹화 저장 위치
- 휘발 대상 영상 선정

**결정 근거**

보안 문제가 가장 큰 리스크이므로 우선 해결 필요

### 안건 3: 화상 기술 스택

**논의 배경**

화상회의 구현을 위한 라이브러리/프레임워크 선정

**논의 내용**

- **OpenVidu 2**: 레퍼런스 다수, 안정적
- **OpenVidu 3**: 레퍼런스 적음, 영어 자료 중심
- 2와 3의 차이: 미디어 서버 변경
- OpenVidu는 포트를 많이 사용 → 다른 서비스 배포 전 먼저 올릴 것
- Nginx 포트 충돌 주의, 리버스 프록시 세팅 복잡
- 이전 AI 화상 프로젝트들 대부분 화상 자체를 어려워함

**결정 사항 (Decision)**

✅ OpenVidu 2.0 사용 결정
✅ OpenVidu 먼저 배포 후 다른 서비스 올리기

**결정 근거**

빠른 개발을 위해 레퍼런스가 많은 OpenVidu 2 선택. 세팅에 시간 낭비하지 않는 것이 중요.

### 안건 4: 실시간 감지 대응 방안

**논의 배경**

실시간 웃음 감지 시 기술적 한계 대비

**논의 내용**

- 프레임 전송 시 대역폭 이슈 예상
- 웃음 감지 리플레이: 바운드 박스 양식 필요 여부
- 실시간이 안 될 경우 차선책:
  - 공격/방어 종료 후 판정 시간에 전송
  - 프레임 낮춰서 경량화 전송
- 미디어 서버로 1차 전송 후 후처리 아키텍처 검토

**결정 사항 (Decision)**

✅ 실시간 우선, 불가 시 준실시간 방식 적용
✅ 대역폭 기술 스택 추가 검토

**결정 근거**

실시간이 안 될 경우를 대비한 차선책 마련 필요

### 안건 5: 프론트엔드 기술 스택

**논의 배경**

사용자 경험 향상을 위한 프론트엔드 기술 검토

**논의 내용**

- 애니메이션 라이브러리: GSAP 권장
- 상태관리 라이브러리 검토 필요
- 프론트 할 일 예상:
  - 승패 관련 효과
  - 시작 전 카운트다운

**결정 사항 (Decision)**

✅ GSAP 애니메이션 라이브러리 검토
✅ 상태관리 라이브러리 서칭

**결정 근거**

게임성 있는 서비스 특성상 시각 효과가 중요
